# 运行时分歧诊断

本文件用于处理“边界已经清楚，但浏览器执行与本地执行不一致”的场景。它关注 first divergence，不负责重新证明请求链，也不替代 `env-patch` 的具体补环境实现。

## 适用条件

进入本文件前应满足：

- 目标请求或目标字段已确认。
- writer / builder / entry / source 至少已有候选闭环。
- 当前问题不是“在哪里生成”，而是“同样输入为什么本地输出不同”。

如果 writer 或 source 仍是猜测，先回到 `request-chain-evidence.md`。

## 分歧分类

运行时问题通常不是单一缺对象，至少分五类：

1. `缺对象`：直接访问 `window`、`document`、`navigator`、`crypto`、`location` 等导致崩溃。
2. `缺状态`：对象存在但状态不对，例如 HttpOnly cookie、上游 seed、storage、session、challenge ticket 缺失。
3. `反调试`：断点、console、堆栈、函数源码、时间差检测导致分支变化。
4. `不稳定源`：时间、随机数、设备种子、序号、生命周期顺序未固定。
5. `风控分支`：浏览器正常态与本地/调试态进入不同风险分支。

补环境前先分类。不要把所有问题都归为“缺 DOM”。

## 诊断顺序

1. 固定输入：同一 payload、同一时间、同一随机数、同一 cookie/storage 摘要。
2. 对齐最终字段：长度、字符集、分段、编码、是否随时间变化。
3. 对齐关键中间值：至少找出第一个不同的中间值。
4. 判断分歧类别：缺对象、缺状态、反调试、不稳定源、风控分支。
5. 只修第一处分歧对应的最小因果单元。
6. 复测并记录分歧是否前移。

## 缺对象 vs 缺状态

`缺对象` 的特征：

- 明确抛错。
- 补一个对象或属性后执行前进。
- 当前链路确实访问该对象。

`缺状态` 的特征：

- 不抛错，但输出错误或进风控。
- 对象表面齐全，状态值缺失或顺序不对。
- 上游响应、cookie、storage、challenge、session 或设备种子恢复后结果明显改善。

HttpOnly cookie、服务器下发 ticket、挑战后二跳 cookie 都是状态，不是简单对象。

## 反调试与观察者效应

出现以下现象时先做干净基线：

- 打开 DevTools 后输出变化。
- 断点命中后进入不同分支。
- 无限 `debugger`。
- 检查函数源码、调用栈、console 行为或时间差。

记录时要区分：

- 调试动作本身造成的变化。
- 调试后才暴露出的真实环境依赖。

## 不稳定源

比较前先固定：

- `Date.now()` / `new Date()`。
- `performance.now()`。
- `Math.random()`。
- 设备种子、安装时间、页面生命周期序号。
- WebSocket / long-polling 序号和 ack。

未固定不稳定源时，任何中间值对比都可能是弱证据。

## 风控分支

风控分支的重点不是“本地输出和浏览器不同”，而是：

- 正常态和风险态第一次在哪里分叉。
- 分叉由哪个状态差异触发。
- 目标 builder 是否在分叉前已完成。
- 风控消费者是否可以绕开，还是必须复现。

不要把风险分支里的输出当成正常算法真值。

## 输出要求

诊断完成后至少写清：

- 当前分歧类别。
- first divergence 位置。
- 浏览器正常态样本来源。
- 本地执行样本来源。
- 需要补的最小状态或对象。
- 下一步动作：补对象、恢复状态、固定不稳定源、隔离反调试、回退请求链或进入验证。

推荐写入：

- `samples/runtime-evidence.jsonl`
- `samples/timeline.jsonl`
- `notes.md` 的“运行时证据与本地重建”部分

## 退出标准

可以退出 runtime 诊断的条件：

- first divergence 已定位。
- 缺对象和缺状态没有混淆。
- 不稳定源已固定或明确列为当前 blocker。
- 下一步补丁或验证动作清楚。
- 相关证据已写入 target artifact。
